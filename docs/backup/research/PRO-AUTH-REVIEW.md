This architecture is solid. A few bolts need tightening, but the core split is the one I would standardize for a reusable template.

The version I would ship is:

* **Supabase Auth** = identity and linked sign-in methods
* **`public.profiles`** = canonical user-facing profile data
* **RevenueCat** = canonical client-side entitlement state
* **`SessionRepository`** = read-only aggregate for the current principal
* **separate command services/repos** for auth actions, profile writes, purchases, switching, and deletion

That lines up well with Supabase’s model: they recommend a public profile table keyed 1:1 to `auth.users.id`, anonymous users still use the `authenticated` role, and turning an anonymous user into a permanent one is done by linking an identity to that same user. RevenueCat likewise supports custom App User IDs, user switching with `logIn()`, and warns that anonymous RevenueCat IDs are created when you configure without an App User ID or call `logOut()`. ([Supabase][1])

The one thing I would actively change from your draft is **profile creation**. Make the `profiles` row a **database invariant**, not just an app behavior. Supabase documents the exact pattern: `public.profiles(id uuid references auth.users on delete cascade primary key, ...)` plus a tiny `auth.users` trigger that inserts the shell row on user creation. Keep the trigger minimal because Supabase also notes that a failing trigger can block signups. App-side `ensureProfile()` can still exist as a repair path, but the trigger should be the source of truth. ([Supabase][1])

### 1. Does the overall model make sense?

Yes. It avoids identity spaghetti.

I would make one small extension: if you ever need **server-side** premium checks inside Edge Functions or database/RLS logic, add an optional backend-owned table such as `account_access` or `subscription_status`, updated from RevenueCat webhooks or fetched from RevenueCat’s REST API. For the **client UI**, RevenueCat `CustomerInfo` is enough. For the **backend**, a mirrored access record is often the clean bridge. RevenueCat explicitly positions webhooks and the REST API as the way to maintain your own backend record of subscription state. ([revenuecat.com][2])

### 2. Should profile data live only in `profiles`?

Yes, for fields like `firstName`, `lastName`, avatar, onboarding flags, and similar user-facing data, `profiles` should be the **canonical** home.

Supabase’s Auth schema is not exposed through the autogenerated API, which is one reason they recommend a public user table with RLS. Also, `user_metadata` is editable by the user and Supabase explicitly says not to use it in security-sensitive contexts. That makes Auth metadata a poor canonical profile store. I would use Auth metadata only for **bootstrap/import** purposes, such as copying an OAuth provider’s initial name/avatar into `profiles` when the row is first created. After that, `profiles` wins. ([Supabase][1])

### 3. Best reactive composition for the session layer?

Your instinct is right. I would compose it like this:

```dart
final principal$ = authRepo.watchPrincipal(); // seeded from currentSession

final userScopedSession$ = principal$
    .distinct((a, b) => a?.stableKey == b?.stableKey)
    .switchMap((p) {
      if (p == null) {
        return Stream.value(SessionState.signedOut());
      }

      return Rx.combineLatest2(
        profileRepo.watchProfile(p.userId),
        billingRepo.watchEntitlement(p.userId),
        (profile, entitlement) => SessionState.ready(
          userId: p.userId,
          isAnonymous: p.isAnonymous,
          profile: profile,
          entitlement: entitlement,
        ),
      ).startWith(
        SessionState.loading(
          userId: p.userId,
          isAnonymous: p.isAnonymous,
        ),
      );
    });

final appState$ =
    Rx.combineLatest2(userScopedSession$, connectivityRepo.watch(), AppState.new);
```

Why this shape: Supabase Flutter v2 expects you to use `currentSession` at startup and then react to auth changes afterward; Supabase table `stream()` gives you the initial row plus later realtime changes; RevenueCat entitlement state should be seeded from `getCustomerInfo()` and then updated via its listener plus foreground refreshes, because RevenueCat’s listener is not a full server-push stream. ([Supabase][3])

### 4. Is `switchMap` the right operator?

Yes, for **user-scoped** downstreams it is exactly the right operator.

When the auth principal changes, the old profile stream and old entitlement stream must die immediately. `switchMap` gives you that without manual disposal gymnastics. I would keep **connectivity outside** that `switchMap`, because connectivity is not tied to a specific user. That keeps your stream graph cleaner and avoids rebuilding user-scoped subscriptions every time the network status flickers.

### 5. Should “upgrade guest to regular account” keep the same Supabase user?

Yes. I would treat that as a hard rule.

Supabase’s anonymous-auth docs describe converting an anonymous user to a permanent user by **linking an identity to the existing user**. Email/phone is done with `updateUser()`, and adding a password requires the email/phone to be verified first. OAuth linking is done with `linkIdentity()`, and manual linking must be enabled in project settings. This is the best long-term model because row ownership, profile row, and RevenueCat customer identity all stay anchored to the same `user.id`. ([Supabase][4])

One nuance that matters a lot: for Google/Apple **upgrade**, call `linkIdentity()` while the anonymous user is signed in. Do **not** use the normal “sign in with provider” method for this path, because that is a login/switch flow, not an upgrade flow. Supabase also notes that `linkIdentity()` fails if that identity is already linked elsewhere, which is exactly the behavior you want: that case should fall into **account switching**, not hidden merging. ([Supabase][5])

### 6. Should “log in to existing account” while anonymous be switching, not merging?

Yes. In the base template, absolutely yes.

Supabase explicitly frames the anonymous-to-existing-account case as one that requires **manual conflict resolution** and possibly manual reassignment of rows, depending on the app’s data model. That is a giant neon sign that says “do not build generic auto-merge into the template.” Also, anonymous users have **no way to sign back in as the same user after sign-out**, so a careless switch can orphan guest data behind an unrecoverable account. ([Supabase][4])

My default template policy would be:

* If the guest has **no meaningful cloud data** and **no active entitlement**, allow switching with a clear confirmation.
* If the guest has **meaningful cloud data** or **active Pro**, steer hard toward **Upgrade this guest account** instead.
* Do **not** auto-migrate guest-owned DB rows in the template.
* If a specific app later needs merging, build it as an explicit backend workflow such as `mergeAnonymousIntoExistingAccount()`, with domain-specific rules per table.

That keeps the default behavior predictable instead of magical and dangerous.

### 7. Should RevenueCat use Supabase `user.id` from the very beginning, even for anonymous users?

Yes. For your exact flow, this is the cleanest pattern.

RevenueCat recommends custom App User IDs when you already have your own user identifiers, and recommends those IDs be unique and non-guessable. Supabase user IDs are UUIDs, so they fit nicely. RevenueCat also says anonymous RevenueCat IDs are created when you configure without an App User ID, and that `logOut()` creates a new anonymous RevenueCat ID too. Since a Supabase anonymous user already has a stable `user.id`, you do **not** need RevenueCat’s own anonymous ID layer here. ([revenuecat.com][6])

My integration rule would be:

* Do **not** configure RevenueCat until you have a Supabase user.
* On first anonymous or regular auth, configure RevenueCat with `appUserID = supabaseUser.id`.
* On account switch, call `Purchases.logIn(newUserId)` directly.
* Do **not** call RevenueCat `logOut()` if you want to stay in custom-ID-only mode. RevenueCat explicitly says switching accounts can go straight to `logIn()` without a preceding `logOut()`. ([revenuecat.com][7])

### 8. What if a guest buys Pro, then tries to log in to an existing account?

This is the sharp edge, and I would make the template opinionated here.

My recommendation: **Pro stays with the current guest account unless the user upgrades that same account.** Logging in to an existing account should show the existing account’s data and entitlements only. Do **not** silently move Pro during a normal login switch. ([revenuecat.com][8])

The UX I would ship:

* Interstitial before switch:

  * “This guest account has data/Pro.”
  * Primary CTA: **Upgrade this guest account**
  * Secondary CTA: **Switch accounts instead**
  * Explicit warning that switching does not carry guest-owned data/Pro to the other account by default

Why so strict: RevenueCat restore behavior can **transfer** or **alias** purchases across App User IDs depending on project settings, and `restorePurchases()` / `syncPurchases()` are exactly the operations that can trigger that movement. RevenueCat also says `restorePurchases()` should only be user-triggered, and `syncPurchases()` carries a risk of transferring or aliasing anonymous users. So the template should **not auto-restore or auto-sync on account switch**. ([revenuecat.com][9])

### 9. Is it reasonable for `SessionRepository` to be the single source of truth for auth status, anonymity, profile, entitlement, onboarding readiness, display name, and tier?

Yes, **as a read model**.

That is not too much responsibility if it only **projects** the current-user state. I would let it expose:

* auth status
* `userId`
* `isAnonymous`
* `profile`
* `entitlement`
* derived `displayName`
* derived onboarding status
* derived user tier

What I would not let it own: sign-in commands, upgrade commands, purchase commands, restore commands, account deletion, or merge/switch logic. Those belong in specialized repos/services. Connectivity is the one field I would keep outside unless you rename the aggregate to something broader like `AppStateRepository`.

### 10. Final recommended architecture and flow

This is the version I would hand to agents:

1. **Database**

   * `public.profiles(id uuid primary key references auth.users on delete cascade, ...)`
   * RLS enabled
   * minimal trigger on `auth.users` insert that creates the shell profile row
   * tables guests may use: normal own-row policies
   * tables permanent users only may use: add restrictive checks on `is_anonymous = false` because anonymous users also authenticate as `authenticated` users in Supabase ([Supabase][1])

2. **AuthRepository**

   * `watchPrincipal()`
   * `continueAsGuest()`
   * `loginExistingWithPassword()`
   * `loginExistingWithOtp()` with `shouldCreateUser: false`
   * `loginExistingWithOAuth()`
   * `upgradeAnonymousWithEmail()` using `updateUser(email)` then verification then `updateUser(password)`
   * `linkProviderToCurrentUser()` using `linkIdentity()`
   * `signOut()`
   * `deleteCurrentAccount()` via server function, not directly from the client ([Supabase][10])

3. **ProfileRepository**

   * `watchProfile(userId)` via Supabase `stream()`
   * `updateProfile(...)`
   * canonical store for first name, last name, avatar, onboarding flags, etc. ([Supabase][11])

4. **SubscriptionRepository**

   * attach RevenueCat to the current Supabase `user.id`
   * seed entitlement with `getCustomerInfo()`
   * listen for `CustomerInfo` updates
   * refresh on app foreground and after purchase/restore/login
   * never use RevenueCat anonymous IDs in this template
   * no `logOut()` in RevenueCat when switching accounts ([revenuecat.com][12])

5. **SessionRepository**

   * single aggregated stream
   * `switchMap` on stable auth principal
   * combine profile + entitlement
   * derive display name, onboarding readiness, and tier
   * keep connectivity in a sibling repo and combine one level higher

6. **Account switching policy**

   * guest → existing login = switch, not merge
   * no automatic DB row migration in template
   * no automatic purchase restore/sync on switch
   * when guest has data or Pro, default CTA should be **Upgrade current account** instead ([Supabase][4])

7. **Delete account**

   * put it in Profile for all account types, including anonymous, to satisfy App Store requirements for apps that support account creation
   * execute deletion server-side because Supabase admin delete requires a `service_role` key and should only be called on the server
   * delete owned Storage objects first, then auth user, then client sign-out
   * remember that deleting `auth.users` does not automatically sign the client out ([Apple Developer][13])

8. **Operational safeguards**

   * enable CAPTCHA / Turnstile on anonymous sign-in
   * add a cleanup policy for stale anonymous users, because Supabase says automatic cleanup is not currently available
   * exclude any anonymous users with active paid entitlements or meaningful retained data from cleanup rules ([Supabase][4])

The three guardrails I would bake into the template, in bright paint, are:

* **Welcome never creates accounts.** If “log in to existing account” ever uses OTP/magic link, `shouldCreateUser` must be `false`, or Supabase will create a user by default. ([Supabase][10])
* **Upgrade current guest never creates a second user.** It must use `updateUser()` or `linkIdentity()`. ([Supabase][4])
* **Switch account never auto-merges data or purchases.** That needs explicit product-specific logic. ([Supabase][4])

So my final verdict: **yes, your model is right**, and I would standardize it with four non-negotiables:

* canonical profile lives in `profiles`
* guest upgrade is in-place identity linking on the same Supabase user
* RevenueCat uses Supabase `user.id` from the first anonymous session onward
* existing-login from a guest is switch-account behavior, not hidden merge behavior
